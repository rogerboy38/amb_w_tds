3. Architecture — How Routing Works
text
User types "@ai create work order for 0307 qty 150"
    ↓
Raven Hook → handle_raven_message() in agent.py (line ~1180)
    ↓
Intent Detection (keyword matching, NOT regex):
  - mfg_keywords: "work order", "bom", "production", "manufacture", "wip" → ManufacturingAgent
  - pay_keywords: "payment", "pay", "invoice", "reconcil", "outstanding" → PaymentAgent  
  - orch_keywords: "workflow", "orchestrat", "end to end", "full process" → WorkflowOrchestrator
  - Default fallback → existing sales_order_bot / RaymondLucyAgent
    ↓
Agent processes request → calls frappe.get_doc/frappe.new_doc → responds in Raven chat
CRITICAL: ALL bots triggered via @ai mention only. No separate Raven Bot entries for manufacturing/payment/workflow. The registered Raven Bots are: Executive Insights, sales_order_follow_up, rnd_bot, sales_order_bot.

8. Next Steps — Intelligence Upgrade (Research Phase)
User wants the agent to be SMARTER:

Ask clarifying questions when it has doubts (not just fail silently)

Use conversational threads for multi-step interactions

Compare to OpenClaw framework architecture

Raven v2 Native Capabilities (from ravenchat.ai):
AI Agents with threads — maintain context of complex multistep tasks

Dynamic instructions (Jinja templates) — personalize per user/context

Functions — call ERPNext functions to fetch/create documents

Tools — file search, vision, code interpreter

Built-in support for exactly what we need

OpenClaw Architecture (from research):
Gateway + LLM + Tools + Memory + Heartbeat/Cron

Persistent daemon, multi-channel, model-agnostic

Proactive scheduling (heartbeat/cron for autonomous actions)

More complex than what we need — Raven v2 already provides the infrastructure

Key Insight:
We don't need to build OpenClaw. Raven v2's AI Agent framework already supports threads, functions, and tools natively. We should leverage Raven's built-in AI Agent features rather than reinvent the wheel. The path forward:

Register agents as proper Raven AI Agents (not just hook-based routing)

Use Raven threads for multi-turn conversations (clarifying questions, confirmations)

Define functions that agents can call (create_work_order, get_bom, etc.)

Add conversational logic: When agent lacks required info, ASK in the thread instead of failing

10. Important Rules Learned
No import frappe in Server Scripts — causes ImportError

bench restart doesn't reload Python — must use docker restart from host

Git remote is upstream not origin — use git pull upstream main

Regex breaks in JSON — double-escaping kills patterns; use keyword matching instead

No bench build needed for Python-only changes

All bots use @ai — single entry point, internal routing by keywords
