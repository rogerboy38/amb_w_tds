4. ERPNext API Key Learnings
4.1 Tax Configuration
Sales Taxes child table requires mexico_tax_type (value "IVA" for 16% tax) — mandatory custom field

cost_center required on tax rows — default: 25000 - Main - AMB-W

Tax template: Mexico Tax - AMB-W, account: IVA - AMB-W

Tax Type doctype values: IVA, IEPS, ISR, IVA 0%

4.2 API Update Gotchas
When updating items via PUT, must send ALL existing fields or ERPNext resets missing ones

Broken Address links cause LinkValidationError — must clear them before saving

InvalidQtyError if qty fields are missing or zero

4.3 IVA Pattern (13.8%)
Formula: QTN_total * 1.16 = SO_total (i.e., QTN has no IVA, SO has 16% IVA added)

This was the most common discrepancy pattern (45 out of 71 cases)

11. Patterns & Lessons Learned
Migration Patterns
IVA 16% pattern: Most common issue — QTN without IVA vs SO with IVA. Formula: QTN * 1.16 ≈ SO

False positives from 2025 QTNs: QTN numbering SAL-QTN-2024-00001 through 00019 were actually 2025 quotations incorrectly matched to old migrated SOs

Address naming: Recurrent error from migration tool where addresses get named as customer instead of "AMB WELLNESS"

ERPNext API Patterns
Always send complete item data on PUT (ERPNext resets missing fields)

Clear broken address links before saving

Tax rows need both mexico_tax_type and cost_center

Use /api/resource/Sales Order/{name} for CRUD operations

Authentication: token-based with api_key:api_secret

Debugging Approach
Three-round error resolution is normal: fix one error, reveal the next

Always check what ERPNext actually saved vs what was sent

Grand total recalculation happens server-side after item/tax changes
