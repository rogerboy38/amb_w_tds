3. Detailed Tasks
Task 7.1 – Mesh Size Parsing for Powders
Goal: Support inputs like 0307 200:1 powder, 100 mesh or Highpol 20/25 100 mesh and capture mesh size.

Implementation:

Model / data_contracts

Add optional field mesh_size: Optional[str] to ParsedSpec and propagate to PlannedItem/PlannedBOM if needed.
​

Parser – detection & normalization

In ProductSpecificationParser (parser.py), add constant list MESH_PATTERNS, e.g.:

r"\b(\d+)\s*mesh\b"

r"\bmesh\s*(\d+)\b"

Normalize to "{number}M" (e.g. 100M).

Add _extract_mesh_size(text: str, family: str) -> Optional[str] that:

Only runs for powder families: 0307, HIGHPOL, ACETYPOL.
​

Checks patterns in order; returns first valid normalized value.

Valid values list can be configurable per family (e.g. 60, 80, 100).

Integration into parsing flow

Call _extract_mesh_size inside _parse_natural_language() and _parse_item_code() (if text includes mesh).

Attach mesh_size to ParsedSpec and carry through engine to FG item attributes or description.

No breaking change requirement

If no mesh size is mentioned, behavior remains identical to Phase 6.

Task 7.2 – Customer‑Specific Naming Patterns
Goal: Allow FG naming rules per customer without breaking the global pattern:

Config structure

Introduce a simple config (JSON file in templates/ or a Python dict in e.g. config.py) for rules, e.g.:

json
{
  "CUSTOMER_XYZ": {
    "pattern": "{FAMILY}-{CUSTOMER_CODE}-{VARIANT}-{PACKAGING}",
    "customer_code": "XYZ",
    "default_cert": "ORG-EU",
    "default_packaging": "1000L-IBC"
  }
}
Spec input

Extend ParsedSpec to have an optional customer and customer_code.

In _parse_natural_language, detect customer tags like "for customer XYZ" or "for XYZ" (simple regex or keyword map).

Engine naming logic

In engine.py _generate_fg_item() (where cert_names already exist), add a layer:

If spec.customer_code (or spec.customer) is present and a rule exists:

Build FG item_code and item_name using the rule’s pattern.

Else fallback to existing {FAMILY}-{ATTRIBUTE}-{VARIANT}-{PACKAGING} pattern.

Backward compatibility

No change for existing requests without customer info.

If customer specified but no rule exists, use fallback (and log a warning in the report).

Task 7.3 – Batch/Lot Tracking Flags
Goal: Ensure BOM‑generated items are compatible with Batch AMB and batch tracking:

Item flags

In ItemAndBOMService (erpnext_client.py), when creating Items for families 0227, 0307, HIGHPOL, ACETYPOL:

Set has_batch_no = 1 (and any custom fields like amb_requires_batch_tracking = 1 used by Batch AMB).

Report hints

In GenerationReport, add a section batch_tracking summarizing:

Whether the created items are batch‑enabled.

Suggested golden number template (if already defined in your Batch AMB project).

No direct Batch AMB modification

Phase 7 does not modify Batch AMB logic; it only ensures BOM outputs are ready for it.

Task 7.4 – Raven Integration (BOM Skill)
Goal: Allow Raven chat users to call the BOM Agent and get answers with ERPNext links.

Skill in raven_ai_agent

Add a new skill module, e.g. agents/bom_creator_agent.py or similar structure already used.

Skill accepts parameters:

request_text: str

dry_run: bool (default true)

optional customer: str

Skill calls ERPNext via REST or internal Frappe client to:

amb_w_tds.ai_bom_agent.api.create_multi_level_bom_from_spec.

Response format

Return Markdown summary with:

Family, certification, variant, packaging, mesh size (if any).

List of created/impacted Items and BOMs.

ERPNext links (/app/item/<name>, /app/bom/<name>).

Channel integration

Configure Raven triggers so that messages containing e.g. 0227, 0307, HIGHPOL, ACETYPOL, BOM, or create BOM suggest the skill.

Docs

Add a short usage section to docs/AI_BOM_AGENT_ORCHESTRATOR_REPORT.md explaining how to invoke this skill in chat.
